CSS has evolved dramatically over the past few years. Features that once required JavaScript workarounds or complex hacks are now achievable with a few lines of native CSS. This article covers the most impactful modern CSS patterns that have reached solid browser support and are ready for production use.

## Container Queries

For years, responsive design meant responding to the viewport width. Container queries change this by letting elements respond to the size of their parent container instead.

This is transformative for component-based architectures. A card component can now adapt its layout based on the space it actually has, rather than making assumptions about the viewport:

```css
.card-container {
  container-type: inline-size;
  container-name: card;
}

.card {
  display: grid;
  gap: 1rem;
}

@container card (min-width: 400px) {
  .card {
    grid-template-columns: 200px 1fr;
  }
}

@container card (min-width: 700px) {
  .card {
    grid-template-columns: 300px 1fr;
    gap: 2rem;
  }

  .card__title {
    font-size: 1.5rem;
  }
}
```

The beauty of this approach is that the same card component works correctly whether it appears in a narrow sidebar, a main content area, or a full-width hero section. You no longer need to create variants like `CardSmall`, `CardMedium`, and `CardLarge`.

Container query units are equally useful. The `cqw` and `cqh` units let you size elements relative to their container:

```css
.card__image {
  width: 100cqw;
  height: min(200px, 40cqw);
  object-fit: cover;
}
```

## Cascade Layers

Cascade layers give you explicit control over the cascade, which is CSS's mechanism for resolving competing styles. They solve the specificity wars that plague large codebases:

```css
@layer reset, base, components, utilities;

@layer reset {
  *, *::before, *::after {
    box-sizing: border-box;
    margin: 0;
    padding: 0;
  }
}

@layer base {
  body {
    font-family: system-ui, sans-serif;
    line-height: 1.6;
    color: #1a1a1a;
  }

  a {
    color: #2563eb;
    text-decoration-thickness: 1px;
    text-underline-offset: 2px;
  }
}

@layer components {
  .btn {
    display: inline-flex;
    align-items: center;
    padding: 0.5rem 1rem;
    border-radius: 0.5rem;
    font-weight: 500;
  }

  .btn--primary {
    background: #2563eb;
    color: white;
  }
}

@layer utilities {
  .sr-only {
    position: absolute;
    width: 1px;
    height: 1px;
    overflow: hidden;
    clip: rect(0, 0, 0, 0);
  }

  .text-center { text-align: center; }
  .mt-4 { margin-top: 1rem; }
}
```

The layer order at the top determines priority. Utilities always win over components, which always win over base styles. This works regardless of source order or specificity. A simple `.text-center` utility in the `utilities` layer will override a `.card__title` rule in the `components` layer, even though `.card__title` is more specific.

This also works well with third-party CSS. You can import external stylesheets into a low-priority layer:

```css
@layer vendor, app;

@import url('some-library.css') layer(vendor);

@layer app {
  /* Your styles always win over vendor styles */
}
```

## The :has() Selector

The `:has()` selector is arguably the most powerful addition to CSS in years. It lets you style a parent element based on its children, something that was previously impossible without JavaScript:

```css
/* Style a form group when its input is focused */
.form-group:has(input:focus) {
  border-color: #2563eb;
  box-shadow: 0 0 0 3px rgba(37, 99, 235, 0.1);
}

/* Style a card differently when it contains an image */
.card:has(img) {
  grid-template-rows: auto 1fr;
}

.card:not(:has(img)) {
  padding: 2rem;
}

/* Adjust layout when a sidebar is present */
body:has(.sidebar) .main-content {
  max-width: 65ch;
}

body:not(:has(.sidebar)) .main-content {
  max-width: 85ch;
  margin-inline: auto;
}
```

One particularly useful pattern is form validation styling. You can highlight error states at the group level when an input is invalid:

```css
.field:has(input:invalid:not(:placeholder-shown)) {
  .field__label {
    color: #dc2626;
  }

  .field__input {
    border-color: #dc2626;
  }

  .field__error {
    display: block;
  }
}
```

## CSS Subgrid

Subgrid lets child elements participate in a parent's grid layout. This solves the long-standing problem of aligning content across sibling grid items:

```css
.card-grid {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(300px, 1fr));
  gap: 1.5rem;
}

.card {
  display: grid;
  grid-template-rows: subgrid;
  grid-row: span 4; /* title, image, description, footer */
  gap: 0;
}
```

Without subgrid, if one card has a longer title, its description starts at a different vertical position than its siblings. With subgrid, all titles align, all descriptions align, and all footers align, because they share the same row tracks.

## CSS Nesting

Native CSS nesting has arrived, and it works similarly to what preprocessors like Sass have offered for years:

```css
.card {
  background: white;
  border-radius: 1rem;
  overflow: hidden;

  & .card__header {
    padding: 1.5rem;
    border-bottom: 1px solid #e5e7eb;
  }

  & .card__title {
    font-size: 1.25rem;
    font-weight: 600;

    & a {
      color: inherit;
      text-decoration: none;

      &:hover {
        color: #2563eb;
      }
    }
  }

  &:hover {
    box-shadow: 0 10px 15px -3px rgba(0, 0, 0, 0.1);
  }

  @media (prefers-color-scheme: dark) {
    & {
      background: #1e293b;
    }
  }
}
```

This reduces repetition and makes the relationship between selectors clearer. The `&` character represents the parent selector, just like in Sass.

## View Transitions API

The View Transitions API enables smooth animations between page navigations or DOM state changes:

```css
@view-transition {
  navigation: auto;
}

::view-transition-old(root) {
  animation: fade-out 200ms ease-out;
}

::view-transition-new(root) {
  animation: fade-in 200ms ease-in;
}

/* Named transitions for specific elements */
.hero-image {
  view-transition-name: hero;
}

::view-transition-old(hero) {
  animation: scale-down 300ms ease-out;
}

::view-transition-new(hero) {
  animation: scale-up 300ms ease-in;
}
```

For same-document transitions triggered by JavaScript state changes, you can use the `document.startViewTransition()` API:

```javascript
function switchTab(newTab) {
  document.startViewTransition(() => {
    updateActiveTab(newTab);
    updateTabContent(newTab);
  });
}
```

## Color Functions and Color Spaces

CSS now supports wider color gamuts and powerful color manipulation functions:

```css
:root {
  --brand: oklch(60% 0.15 250);
  --brand-light: oklch(from var(--brand) calc(l + 0.2) c h);
  --brand-dark: oklch(from var(--brand) calc(l - 0.2) c h);
  --brand-muted: oklch(from var(--brand) l calc(c * 0.5) h);
}

.btn-primary {
  background: var(--brand);
  color: white;
}

.btn-primary:hover {
  background: var(--brand-dark);
}
```

The `oklch` color space is perceptually uniform, meaning that changes in lightness look consistent across different hues. The `from` keyword enables relative color syntax, letting you derive new colors from existing ones without redefining the whole value.

## Practical Tips for Adoption

Start by using cascade layers in your next project. They require no browser support concerns at this point and immediately improve your CSS architecture. Container queries are the next priority since they fundamentally improve how you build reusable components.

For the `:has()` selector, begin with progressive enhancement. Add `:has()`-based styles that improve the experience when supported, with reasonable fallbacks for the rare browsers that lack support.

These features work well together. You might use cascade layers to organize your styles, container queries to make components adaptive, `:has()` for context-aware styling, and subgrid for consistent alignment. Combined, they represent a fundamental shift in what is possible with CSS alone.
